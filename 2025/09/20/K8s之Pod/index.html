什么是Pod

Pod是kubernetes中最小的可部署单元。一个Pod包含了一个应用程序容器（或者多个容器）、存储资源、一个唯一的IP地址、以及一些确定该容器如何运行的选项。Pod容器组代表了Kubernetes中一个独立的应用程序运行实例。他们：

  • 共享网络命名空间：Pod内所有的容器共享同一个IP地址和端口空间。
  • 共享存储卷：Pod可以定义一组共享的存储卷，使容器间可共享文件
  • 共享进程空间：容器间可以看见彼此的进程
  • 资源共享与隔离：Pod作为整体申请资源，K8s调度器基于Pod的资源需求分配节点

Pod的生命周期一致性

  • Pod内额所有容器同时启动、终止
  • 健康检查以Pod为整体单位

Pod的整体创建流程（原文档：https://www.yuque.com/dycloud/pss8ys/vfu5nhag71ezmbl1）

Pod 生命周期状态详解

状态 触发条件 典型原因 排查方法
**Pending** API Server 已接收创建请求 • 调度队列等待• 镜像正在拉取• 节点资源不足 kubectl describe pod 查看 Events
**ContainerCreating** Kubelet 开始创建容器 • 存储卷挂载中• CNI 网络配置中• CRI 创建容器中 检查节点存储/网络插件日志
**Running** 至少一个容器处于运行状态 正常服务状态 检查业务日志 kubectl logs
**CrashLoopBackOff** 容器反复崩溃（重启间隔指数增长） • 应用启动错误• 配置错误• 内存泄漏导致 OOM• Liveness 探针配置不当 kubectl logs --previous 查看崩溃前日志
**ImagePullBackOff** 容器镜像拉取失败 • 镜像名称错误• 私有仓库认证失败• 网络隔离• 镜像仓库故障 kubectl describe pod 查看拉取错误详情
**Terminating** 删除指令已下发 用户执行删除或控制器缩容 检查 Finalizer 阻塞 kubectl get pod -o json
**Succeeded** 所有容器正常退出（exit code=0） Job/CronJob 任务完成 检查任务输出日志
**Failed** 至少一个容器异常退出（exit code ≠0） • 应用代码异常• 资源超限被杀• 健康检查失败 kubectl describe 查看 Exit Code
**Unknown** Kubelet 失联 • 节点故障• 网络分区• Kubelet 进程崩溃 kubectl get node 检查节点状态
**Evicted** 节点资源压力触发驱逐 • 内存不足• 磁盘压力• PID 资源耗尽 检查节点监控指标 kubectl top node

当一个 Pod 被删除时，执行一些 kubectl 命令会展示这个 Pod 的状态为 Terminating（终止）。 这个 Terminating 状态并不是 Pod 阶段之一。 Pod 被赋予一个可以体面终止的期限，默认为 30 秒。 你可以使用 --force 参数来强制终止 Pod

Pod 重启策略

Pod 的重启策略通过 spec.restartPolicy 字段定义，控制容器异常退出后的恢复行为：

策略 触发场景 典型应用场景 容器行为
Always 容器退出（无论exit code值） Web服务、数据库等长期运行应用 无限重启容器（Kubelet控制，30秒指数退避重试）
OnFailure 容器以非0状态退出（exit code ≠0） 批处理任务、CI/CD流水线任务 按需重启，成功后停止
Never 任何退出都不重启 单次执行任务（如数据迁移） 记录错误日志，Pod状态变为 Failed

关键约束：

  1. 节点级操作：重启由 Kubelet 执行（不经过API Server）
  2. 避退延迟：防止频繁崩溃消耗资源（第N次重启延迟 = min(300s, 10s * 2^(N-1))）
  3. 重置条件：容器持续运行超过 10分钟 后重启计数归零

Pause 容器

Infra 容器（Pause容器）是Pod的核心基础设施，使用官方 registry.k8s.io/pause 镜像（约250KB）。

功能 实现原理 对业务容器影响
共享网络命名空间 创建初始网络栈，后续容器通过 --net=container:<infra> 加入 容器间通过 localhost 通信
共享存储卷 首先挂载Pod声明的Volume，后续容器复用挂载点 容器共享同一目录数据
共享IPC命名空间 设置 --ipc=container:<infra> 容器可用SystemV IPC通信
PID资源托管 作为PID 1进程（防止僵尸进程） 业务进程退出时被pause回收
Pod生存状态基准 Kubelet通过监控pause容器判断Pod存活 即使业务容器崩溃Pod仍被视为存在

Pod数量限制

Kubernetes Node节点每个默认允许最多创建110个Pod，有时可能会由于系统硬件的问题，从而需要控制Node节点的Pod的运行数量。

即：需要调整Node节点的最大可运行Pod数量。

一般来说，我们只需要在 kubelet 启动命令中增加 –max-pods 参数，然后，重启 kubelet 服务，就生效。

重启 kubelet，不影响现有运行中的容器，不会造成容器重启。

Pod解析

某些情况下，DNS 或者其他的域名解析方法可能不太适用，您需要配置 /etc/hosts 文件，在Linux下是比较容易做到的，在 Kubernetes 中，可以通过 Pod 定义中的 hostAliases 字段向 Pod 的 /etc/hosts 添加条目。

适用其他方法修改 Pod 的 /etc/hosts 文件是不被推荐的，因为 kubelet 可能在重新创建 Pod 时，就会覆盖这些修改。

Pod网络策略

如果你希望在 IP 地址或端口层面控制网络流量，可以考虑为集群中特定应用使用网络策略（NetworkPolicy）

Pod 时可以通过以下三个标识符的组合来判别是否可以通讯

  • 其他被允许的 Pods（例外：Pod 无法阻塞对自身的访问）
  • 被允许的名字空间
  • IP 组块（例外：与 Pod 运行所在的节点的通信总是被运行的，无论 Pod 或节点的 IP 地址）

网络策略通过网络插件来实现。这意味着你使用的网络插件是支持 NetworkPolicy 的。

默认情况下，所有的连接都是被允许的
